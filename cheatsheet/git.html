command note
$ sudo apt-get install git -installed git
$ git --version -made sure that git was correctly installed
$ git config --global user.email rami.daoud@powercoders.org -Configuring Git with a couple of pieces of information it's going to need later. (step1)
$ git config --global user.name "Rami Daoud" -Configuring git (step2)
$ cd poco -Navigate to the poco directory
-Git needs to know where the top, or root, of the repository is going to be. Once you do this Git can start to manage all the files in that directory, and all the directories below.
-You can have multiple Git repositories on your computer, as long as they are not inside one another.
$ pwd -Make sure you're in the correct directory with pwd
$ git init -We have to tell Git where the "root" of the repository is by git init,But don't put a repository inside another
$ git status -Checking the state of the repository
$ cd scratch -Moving down to scratch directory to experiment
$ git add index.html -Adding the scratch/index.html file to the index.
$ git status -Now Git is telling us that there are changes to be committed to the repository. These changes have not yet been committed, we still need to do that, which is the next step.
-Remember, this file has been copied to the index. The git rm example it's showing is the command you would run if you changed your mind, and wanted to remove the file from the index.
-This wouldn't delete it from the working directory, just the index.
$ git commit -m "save first version of file." -To make a copy of the file, or files, in the index, and save them in the repository we use git commit.
-Git needs us to give a message that is associated with the commit.
-Here, I'm using the -m flag to specify the commit message.
-If you don't do that then Git will open a text editor to allow you to type in a message.
-Don't forget the double quotes.
$ git status -What do you see?
$ git log -Shows you what's changed
-Edit scratch/index.html. Add anything you like
$ git diff -This shows the difference between what's in the working directory and what's in the index. and if the index is empty then it compares to the copy in the repository
This command is optional like the git status
$ cd cheatsheet -Navigating to the directory we are about to add its files to the index
$ git status -Now Git is telling us that there are changes to be committed to the repository. These changes have not yet been committed, we still need to do that, which is the next step.
$ git add html.html index.html git.html shell.html -Adding the certain files in the cheatsheet directory to the index.
$ git status
$ git diff --cached This helps if you want to compare the working directory with the copy in the index
$ git commit -m "start my cheatsheet"
$ git log
$ cd .. -Moving up to the poco directory (is this necessary for the next step? yes better be in the root before you add the origin)
$ git remote add origin https://github.com/RamiDaoud1979/powercoders.git -We do this step after we already created a repository on Github website
-git remote add command tells your local repository about remotes (copies of your repository that are stored else where)
-This is also the URL for the repository in the browser, with .git added to the end, so you can cut and paste.
-The origin in the command is the short-name that we're going to use for the remote.
-You can use anything you want. However, the convention in Git is that the public version of the repository is called origin
$ git remote -v To view configured remotes
$ git push -u origin master
$ git push 		-Now that your local repository is configured to sync with the repository on GitHub we can copy all of our commits so far to the copy of the repository on GitHub.
-To do that, run the git push command.
-The very first time we do this we have to tell Git exactly how these two repositories are linked, including how they branch.
-You will probably be prompted for the password for your GitHub account.
